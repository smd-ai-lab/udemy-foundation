🏗️ La philosophie d'OpenSpec

OpenSpec se distingue par plusieurs principes fondamentaux :

"Brownfield-first"

Contrairement à d'autres outils comme Spec Kit de GitHub (optimal pour les projets "greenfield" de zéro), OpenSpec est conçu pour les projets réels avec du code existant.

Pourquoi c'est important ? La plupart des développeurs ne partent jamais de zéro. Vous devez modifier, étendre, refactoriser du code qui existe déjà. OpenSpec excelle dans ce cas d'usage "1→n" (ajouter à l'existant) plutôt que "0→1" (créer depuis rien).

Fluide, pas rigide

OpenSpec rejette l'approche waterfall : vous n'êtes pas enfermé dans des phases rigides. Vous pouvez :

  • Modifier n'importe quel artefact à tout moment

  • Itérer sur les specs avant et pendant l'implémentation

  • Revenir en arrière si nécessaire

📁 La structure de projet OpenSpec

Voici comment OpenSpec organise votre projet :

text

mon-projet/

├── openspec/

│   ├── specs/              # 📌 SOURCE DE VÉRITÉ

│   │       └── spec.md     # "Ce que le système fait MAINTENANT"

│   ├── changes/            # 🔄 Changements en cours

│   │   ├── first-version/  # Un changement spécifique

│   │   │   ├── proposal.md # Pourquoi ? (le problème à résoudre)

│   │   │   ├── design.md   # Comment ? (approche technique)

│   │   │   ├── tasks.md    # Quoi ? (liste d'implémentation)

│   │   │   └── specs/      # Delta : ce qui va changer

│   │   │           └── spec.md  # "Ce qui sera AJOUTÉ/MODIFIÉ"

│   │   │

│   │   └── archive/        # ✅ Changements terminés

│   │

│   └── project.md          # Contexte global du projet

3.1 La distinction clé : specs/ vs changes/

  • openspec/specs/ : la source de vérité - ce que le système fait réellement

  • openspec/changes/ : les propositions - ce que vous voulez changer

Cette séparation est fondamentale : elle permet de savoir exactement ce qui est actuel vs ce qui est en cours.

3.2 Le format Delta

Quand vous proposez un changement, le fichier spec.md dans changes/ ne contient pas la spec complète, mais uniquement un delta (un "patch") :

markdown

# Delta for Auth

## ADDED Requirements

### Requirement: Two-Factor Authentication

The system MUST require a second factor during login.

#### Scenario: OTP required

- WHEN a user submits valid credentials

- THEN an OTP challenge is required

## MODIFIED Requirements

### Requirement: Session Management

[description de ce qui change...]

Pendant l'archivage, OpenSpec fusionne automatiquement ce delta dans la spec principale.

🔄 Le workflow complet

Étape 1 : Démarrer un projet

bash

# Installation

npm install -g @fission-ai/openspec@latest

# Dans votre projet

cd llm-leaderboard

openspec init

L'initialisation vous guide pour :

  1. Choisir votre assistant IA (Claude Code, Cursor, Copilot...)

  2. Créer la structure openspec/

Astuce : Remplissez openspec/project.md avec le contexte de votre projet (tech stack, conventions, objectifs). L'IA l'utilisera pour générer des specs cohérentes.

Étape 2 : Explorer une idée

bash

/opsx:explore Build an application which get data from SOURCES.md relative to LLM models performance. Persist data in light sql database, with a date DD/MM/YYYY hh:mm:ss. We add sources furthermore, the application will be able to add new source without losing information. No need UI or backend service for this version first version 0.0.1. It's only get a source, persist. When adding a new source, get it persist it. And so on we could add whatever kind of source based on URL web site for this version.

Cette commande permet d'explorer, poser des questions, inspecter le code existant sans créer de fichiers. C'est utile quand l'idée n'est pas encore claire.

Étape 3 : Proposer un changement

bash

/opsx:propose first-version

L'IA génère automatiquement :

  • proposal.md : pourquoi ce changement est nécessaire

  • design.md : l'approche technique (optionnel)

  • tasks.md : la checklist d'implémentation

  • specs/ : les deltas de spécifications

Étape 4 : Valider et itérer

bash

# Vérifier la structure

openspec validate first-version

# Voir ce qui a été généré

openspec show first-version

L'IA met à jour les fichiers en conséquence. Vous pouvez itérer autant que nécessaire.

Étape 5 : Implémenter

bash

/opsx:apply

L'IA suit tasks.md étape par étape, en cochant les cases au fur et à mesure. Elle ne touche que ce qui a été planifié.

Étape 6 : Archiver

bash

/opsx:archive

Cette commande :

  1. Fusionne les deltas dans openspec/specs/

  2. Déplace le dossier vers changes/archive/

  3. Met à jour la source de vérité
